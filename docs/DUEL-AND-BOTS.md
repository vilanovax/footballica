> مخاطب: تیمِ توسعه. هدف: هرکس بدونِ خواندنِ کل کد بفهمد دوئل و ربات چطور
> کار می‌کنند، چطور تستشان کند، و از کجا توسعه‌شان دهد.

مسیر کد: `backend/src/duel/` و `backend/src/bots/`.

---

## ۱) دوئل چیست و چه قوانینی دارد

دوئل، مسابقهٔ **۱به۱ و ناهم‌زمان (async)** است. دو بازیکن به **یک مجموعهٔ ثابتِ
یکسان** از سؤال‌ها جواب می‌دهند؛ لازم نیست هم‌زمان آنلاین باشند. سرور امتیازها را
مقایسه و برنده را تعیین می‌کند.

سه قانونِ غیرقابل‌نقض که رعایت شده‌اند:

1. **async-first:** بدون Socket.IO. فقط REST. هر بازیکن لِگِ خودش را هر وقت خواست
   بازی می‌کند.
2. **server-authoritative:** تایمر (`deadlineAt`) و درستی فقط سمتِ سرور. فیلدِ
   `isCorrect` هرگز در پاسخِ راند به کلاینت نمی‌رود؛ گزینهٔ درست فقط بعد از ثبتِ
   پاسخ برمی‌گردد.
3. **دوئلِ برابر (ضدِ pay-to-win):** قدرتِ باشگاه/ارتقاها روی دوئل اثر ندارد. فقط
   دانش و سرعت. (وقتی اقتصادِ باشگاه اضافه شود هم همین می‌ماند.)

## ۲) مدل داده (Prisma)

```
Match (mode = DUEL)
 ├─ status: WAITING → ACTIVE → FINISHED         (enum MatchStatus)
 ├─ totalRounds: تعداد سؤال (پیش‌فرض ۵)
 ├─ MatchPlayer[]   ← یک ردیف به‌ازای هر بازیکن (score, isWinner)
 ├─ DuelQuestion[]  ← مجموعهٔ ثابتِ سؤال‌ها (order 0..N-1)  ← «انصاف»
 └─ DuelRound[]     ← راندِ هر بازیکن روی هر سؤال (تایمرِ مستقل)
```

- **`DuelQuestion`**: در لحظهٔ ساختِ دوئل، N سؤالِ تأییدشدهٔ تصادفی انتخاب و اینجا
  ثابت می‌شوند. هر دو بازیکن دقیقاً همین‌ها را می‌بینند → مقایسه منصفانه است.
- **`DuelRound`**: چون async است، هر بازیکن برای هر سؤال `startedAt`/`deadlineAt`
  **مستقلِ خودش** را دارد. `isCorrect`, `points`, `msTaken`, `answeredAt` را سرور
  پر می‌کند. یکتا: `(matchId, userId, order)`.

چرا جدولِ جدا به‌جای استفاده از `MatchRound` تک‌نفره؟ تا موتورِ تک‌نفرهٔ تست‌شدهٔ
فاز ۰ دست‌نخورده بماند و منطقِ دو مدل قاطی نشود.

## ۳) API دوئل

همه پشتِ `JwtAuthGuard`‌اند؛ `userId` از توکن می‌آید (نه از body).

| متد | مسیر | کار |
|---|---|---|
| POST | `/duels/find` | مچ‌میکینگ: پیوستن به دوئلِ باز یا ساختِ دوئلِ منتظر |
| POST | `/duels/:id/next-round` | راندِ بعدیِ همین بازیکن (شروع/ادامه) |
| POST | `/duels/rounds/:roundId/answer` | ثبتِ پاسخ (درستی/امتیاز سمت سرور) |
| GET | `/duels/:id` | وضعیتِ دوئل برای این بازیکن |
| GET | `/duels` | فهرستِ دوئل‌های من (برای صفحهٔ خانه) |

### نمونهٔ چرخه
```
POST /duels/find            → { matchId, status: "waiting"|"matched", totalRounds }
POST /duels/:id/next-round  → { finishedLeg:false, round:{ roundId, order, question,
                                 options[], deadlineAt, serverNow } }   ← بدون isCorrect
POST /duels/rounds/:rid/answer  body { optionId | null }
     → { isCorrect, isLate, points, correctOptionId, msTaken,
         legFinished, matchFinished }
GET  /duels/:id  → { status, totalRounds, me{answered,legFinished,score},
                     opponent{name,answered,score|null}, result|null }
```

- امتیازِ حریف (`opponent.score`) تا وقتی دوئل **FINISHED** نشده `null` است.
- `result.outcome`: `"win" | "loss" | "draw"`.

### امتیازدهی
مثل «بازی سریع»: جوابِ درست = ۱۰۰ + تا ۵۰ جایزهٔ سرعت (خطی، نزولی نسبت به زمان).
غلط/دیر = ۰. ثابت‌ها بالای `duel.service.ts`.

## ۴) مچ‌میکینگ (جریان کامل)

```
POST /duels/find
  │
  ├─ آیا خودم یک دوئلِ WAITING دارم؟ ───────────► همان را برگردان (waiting)
  │
  ├─ آیا دوئلِ WAITINGِ حریفِ دیگری هست؟ ────────► به آن بپیوند (matched) [انسان‑با‑انسان]
  │
  └─ نه ► دوئلِ WAITING بساز + N سؤالِ ثابت ────► (waiting)
```

**بازیکن منتظرِ حریف نمی‌ماند تا بازی کند** — چون async است، بلافاصله لِگِ خودش را
می‌زند (`next-round` در حالتِ WAITING هم مجاز است).

**پرکردن با ربات** وقتی رخ می‌دهد که بازیکن لِگش را **تمام کند** ولی هنوز حریفی
نپیوسته باشد (`finalizeOrFillBot` در `duel.service.ts`):
- اگر حریفِ انسانی پیوسته بود → منتظرِ اتمامِ او می‌مانیم، بعد نهایی.
- اگر تنها بودیم → یک ربات وارد، لِگش شبیه‌سازی، و دوئل **بلافاصله** نهایی می‌شود.

نتیجه: بازیکن‌های واقعی همیشه به هم اولویت دارند؛ ربات فقط وقتی می‌آید که کسی
نبود. (توضیحِ توسعه: می‌شود یک «تأخیرِ انتظار برای انسان» با BullMQ اضافه کرد تا
حتی بعد از اتمامِ لِگ، چند ثانیه منتظرِ حریفِ انسانی بماند.)

## ۵) سیستم ربات

هدف: بازیکن **نفهمد** با ربات بازی می‌کند.

### ربات چیست
یک رکوردِ `User` معمولی با `isBot = true` و `botDifficulty`. نام/آواتار دارد تا مثل
بازیکن دیده شود. این دو فیلد **فقط سمتِ سرور**اند و در هیچ پاسخی به کلاینت
(`/duels/*`, `/leaderboard`, `/auth/*`) نمی‌آیند.

### درجهٔ سختی (`bots/bot.constants.ts`)
دقت و سرعتِ شبیه‌سازیِ ربات را تعیین می‌کند:

| درجه | احتمالِ درست | زمانِ فکر (ms) | حسِ حریف |
|---|---|---|---|
| EASY | ۴۵٪ | ۶۰۰۰–۱۲۰۰۰ | ضعیف و کند |
| MEDIUM | ۶۵٪ | ۳۰۰۰–۸۰۰۰ | متوسط |
| HARD | ۸۵٪ | ۱۵۰۰–۵۰۰۰ | حرفه‌ای و سریع |

این جدول عمداً در یک فایلِ ثابت جداست تا تیمِ بالانس بدونِ دست‌زدن به منطق آن را
تنظیم کند.

### شبیه‌سازیِ لِگِ ربات (`simulateBotLeg`)
برای هر سؤالِ ثابتِ دوئل:
1. با احتمالِ `pCorrect` گزینهٔ درست، وگرنه یک گزینهٔ غلطِ تصادفی.
2. `msTaken` تصادفی در بازهٔ درجه → امتیازِ سرعت واقع‌گرایانه.
3. یک `DuelRound`ِ پاسخ‌داده‌شده ساخته و امتیاز جمع می‌شود.

⚠️ امتیازِ ربات **وارد لیدربوردِ سراسری نمی‌شود** (بدون `zincrby`) تا رتبه‌بندی از
ربات‌ها پاک بماند و لو نروند.

### مدیریت ربات (پنل ادمین)
پشتِ `AdminGuard` (هدرِ `x-admin-key`):

| متد | مسیر | کار |
|---|---|---|
| GET | `/admin/bots` | فهرست + تعداد دوئل هر ربات |
| POST | `/admin/bots` | ساخت: `{ name, difficulty, avatar? }` |
| PATCH | `/admin/bots/:id` | ویرایش نام/درجه/آواتار |
| DELETE | `/admin/bots/:id` | حذف (فقط اگر دوئلی بازی نکرده باشد) |

## ۷) چطور تست کنیم (دستی)

```bash
# ۱) ربات بساز (ادمین)
curl -X POST localhost:3000/admin/bots -H 'x-admin-key: dev-admin-key' \
  -H 'Content-Type: application/json' -d '{"name":"علی رضایی","difficulty":"HARD"}'

# ۲) ورود بازیکن → توکن
#    /auth/request-otp سپس /auth/verify-otp (کد در حالت dev در پاسخ است)

# ۳) دوئل پیدا کن، لِگت را بزن، وضعیت را بگیر
curl -X POST localhost:3000/duels/find -H "Authorization: Bearer <TOKEN>" \
  -H 'Content-Type: application/json' -d '{"totalRounds":5}'
# سپس برای هر راند: POST /duels/:id/next-round و POST /duels/rounds/:rid/answer
# بعد از راندِ آخر، ربات خودکار وارد و دوئل نهایی می‌شود.
curl localhost:3000/duels/<matchId> -H "Authorization: Bearer <TOKEN>"
```

بررسی‌های کلیدی:
- `round` هرگز `isCorrect` ندارد.
- در پاسخِ `/duels/*` هیچ `isBot`/`difficulty` نیست.
- ربات در `/leaderboard` ظاهر نمی‌شود.

## ۸) نقاطِ توسعه (کارهای بعدی)

- **تطبیقِ درجهٔ سختیِ ربات با سطحِ بازیکن** (الان تصادفی است — `BotsService.pickRandom`).
- **تأخیرِ انسانی برای پاسخِ ربات** با BullMQ تا لِگِ ربات «زنده» به‌نظر برسد.
- **پوش «نوبتِ توست»** (Pushe) وقتی حریف جواب داد یا دوئل آماده شد.
- **هوادار/اقتصاد**: در نهایی‌سازیِ دوئل، جایزهٔ «هوادار» به برنده اضافه شود
  (طبق `ECONOMY.md`).
- **پرچمِ فعال/غیرفعال برای ربات** تا بشود ربات را از مچ‌میکینگ کنار گذاشت بدونِ حذف.
- **رهاکردن/تایم‌اوتِ دوئل** برای دوئل‌های انسانیِ نیمه‌تمام.
